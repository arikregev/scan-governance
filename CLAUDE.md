## Project Overview

This Project is a scan-governance micro-service that documents execution of Temporal Workflows by writing them to a postgres db.
The project will listen to a Kafka topic, each message is an instruction to execute a workflow, scan-governance service will
consume the messages as well and will write them into DB.
The microservice will keep tabs on the scan using Temporal SDK and will update the workflow status until it will reach Completed / Failed or timeout.
all parameters should be configurable.

Tech Stack:
- Java 21
- Maven
- Quarkus 3.31.0
- Kafka
- Postgres
- JDBI

## Architecture

Service named Scan-Service is the producer of the kafka messages, each message is a scan event for temporal to execute.
The messages are OCSF, Scan Activity to be exact, based on Proto.
Scan-governace is writing to a database named Workflow, the service will have to match data received from the OCSF and
commit it into workflow database.

the following is the table definition:
```
CREATE TABLE workflow (
    id UUID PRIMARY KEY,
    request_ref UUID,
    workflow_id VARCHAR(500),
    run_id VARCHAR(255),
    scanning_tool VARCHAR(100),
    scan_type VARCHAR(50),
    status VARCHAR(50),
    result JSONB,
    input JSONB,
    started_at TIMESTAMP,
    completed_at TIMESTAMP,
    duration_seconds INTEGER,
    created_at TIMESTAMP DEFAULT,
    updated_at TIMESTAMP DEFAULT now()
);
```

From OCSF:
scan -> uid is request_id in the workflow table.
metadata -> original_event_uid is the workflow_id itself.

take note that run_id, input, result, completed_at and status should be fetched from temporal using the Temporal SDK.

importent note:
- you can have multiple requests for the same workflow id.
- each request can initiate multiple and disntinct workflows each identifed with it's own workflow_id.
- each workflow_id can have multiple run_ids in the event that a workflow was triggered multiple times. (Scan being initiated multiple times)
- while request_id is being generated by scan-service and run_id is generated by Temporal, we would want to match them exactly.
think about the edge case that scan-governace service was down for multiple hours and in that time there where multiple executions of workflows which caused the creation of multiple run_ids. 
Kafka messages are keyed and will be received in order, deal with the issue.
- When writing initally to DB, using Temporal SDK, try and pull all nesseary data:
 -- If temporal doesn't have data yet, put QUEUED in status.
 -- If Status is running pull start time, Input and status.
 -- When Status is Completed, update that in DB and fetch the Result and completedat from temporal as well. also using both completed and stated at, calculate the duration of the workflow and commit it in the approprate column.

 Request table which is read only looks like the following:

for our workflow table, we have the request_id, for request_ref we will need to query request table and find the approprate id (PK) to place in the request_ref.

after initial writing to DB with the appropriate status fetched from Temporal,
the `WorkflowStatusPoller` scheduler runs every N seconds (configurable via `scan.governance.poller.interval`)
and calls `WorkflowGovernanceService.refreshWorkflow()` for each non-terminal row until it reaches a terminal state
or the configurable timeout expires.

one @Inject annotation above constructor and all class variables set with private and final.

### Key design decisions

- **Kafka key = workflow_id**: ordering is guaranteed per partition, so multiple messages for the same workflow are always processed in sequence.
- **Catch-up (service was down, multiple Temporal runs accumulated)**: `syncTemporalRuns` lists *all* Temporal executions for a workflow_id and upserts them. Existing QUEUED placeholders without a run_id are matched to untracked Temporal runs in FIFO order (oldest placeholder → oldest untracked run).
- **run_id is null initially**: a row is inserted with `status=QUEUED` and `run_id=NULL` on first Kafka message; the run_id is filled in by `syncTemporalRuns` once Temporal has created the execution.
- **input / result from Temporal history**: input is extracted from the `WorkflowExecutionStarted` event; result/failure from the close event using `HISTORY_EVENT_FILTER_TYPE_CLOSE_EVENT`. Temporal's default DataConverter stores payloads as raw UTF-8 JSON in `payload.data`.
- **All configuration is externalised** via env vars (see `application.properties`). Key vars: `TEMPORAL_HOST`, `TEMPORAL_NAMESPACE`, `KAFKA_BOOTSTRAP_SERVERS`, `KAFKA_SCAN_EVENTS_TOPIC`, `DB_*`, `POLLER_INTERVAL`, `WORKFLOW_TIMEOUT_HOURS`.
